# Phase E Design

## 1. Scope and sub-phase breakdown

Phase E is the first concrete delivery of Chess Life's hidden-trainer promise. The mechanic is a puzzle-under-pressure loop with two bonus sources that coexist: training bonuses earned out of game, and Flow bonuses earned in game. To keep this shippable and testable, Phase E is split into four sub-phases:

- **E.1 — Puzzle foundation**: `puzzle-data.js`, `puzzle-system.js`, standalone self-training, aptitudes, seen-puzzle tracking, reinforcement queues, Node tests.
- **E.2 — In-game training bonus path**: training-bonus inventory, invoke-from-game, suspend real game, temporary puzzle board, resolve, Stockfish playback reward.
- **E.3 — Coach + finance layer**: `coach-data.js`, `staff-system.js`, single coach slot, weekly cost, coach-browser UI, coach quality applied to sessions and rewards.
- **E.4 — Flow integration**: gain one Flow bonus per Flow entry, hidden-theme puzzle reveal, unseen-puzzle invariant, reveal card, bonus events finalized.

This document goes deep only on **E.1**. E.2-E.4 are defined here only at scope level.

## 2. Locked decisions

- One coach slot only. Coach changes never wipe player learning state.
- Coach skill profile covers exactly 22 themes:
  `fork`, `pin`, `skewer`, `discoveredAttack`, `hangingPiece`, `sacrifice`, `trappedPiece`, `attackingF2F7`, `mateIn1`, `mateIn2`, `backRankMate`, `opening`, `middlegame`, `endgame`, `deflection`, `attraction`, `ruyLopez`, `sicilianDefense`, `frenchDefense`, `caroKannDefense`, `italianGame`, `queensPawnGame`.
- Reward mapping uses a helper `getCoachMoveBonus(skill)` with locked bands:
  `0..20 => 0`, `21..50 => 1`, `51..80 => 2`, `81..100 => 3`.
- Strict no-floor rule: weak coach on a mismatched theme gives `+0`.
- Reinforcement exit is `N = 2`: two successful confirmations in separate sessions; failure during confirmation resets progress.
- `bonus_invoked` and `bonus_resolved` both exist.
- Flow bonus theme is hidden during the puzzle and revealed after.
- No invocation threshold: the player may spend a bonus at any time on their turn.
- Playback is non-interruptible in Phase E, clicks disabled, Focus paused.

## 3. E.1 detailed plan

### Files to create

- `js/puzzle-data.js`
- `js/puzzle-system.js`
- `tests/puzzle-system.test.js`

E.1 intentionally ships **no dedicated UI screen yet**. The logic must be ready for later UI, but tests and debug hooks carry this first slice.

### `puzzle-system.js` public API

Proposed API:

- `PuzzleSystem.init()`
- `PuzzleSystem.getThemes() -> string[]`
- `PuzzleSystem.getThemeLabel(theme) -> string`
- `PuzzleSystem.getAptitude(theme) -> number`
- `PuzzleSystem.getTrainingBonusCount(theme) -> number`
- `PuzzleSystem.getReinforcementQueue(theme) -> Array<{ puzzleId, state, confirmations }>`
- `PuzzleSystem.startSelfTrainingSession(theme, options = {}) -> session`
- `PuzzleSystem.submitSessionAnswer(sessionId, solved) -> result`
- `PuzzleSystem.completeSession(sessionId) -> summary`
- `PuzzleSystem.getSeenCount() -> number`
- `PuzzleSystem.clearForTests()`

E.1 keeps session interaction deliberately simple: a session can be modeled as a small set of selected puzzles plus pass/fail submission hooks. Real board move validation can wait until the later training UI pass; for E.1, the system should already support puzzle selection, scoring, aptitude growth, seen tracking, and reinforcement mutation.

Boundary rule to repeat in code comments: **E.1 trusts the caller**. `submitSessionAnswer(sessionId, solved)` accepts a boolean decided by tests, debug hooks, or later UI. Move-level validation against FEN/solution is explicitly deferred to E.2.

### `puzzle-data.js` format

Static array, one object per puzzle:

```js
{
  id: 'fork_001',
  fen: '2r3k1/5ppp/3p4/3N4/8/8/5PPP/4R1K1 w - - 0 1',
  solution: ['Ne7+'],
  theme: 'fork',
  difficulty: 1320,
  source: 'lichess:abcd1234'
}
```

E.1 should ship a **real Lichess-backed starter pool**, not hand-written placeholders:

- target size: **176 puzzles total**
- locked size: **8 puzzles per theme × 22 themes = 176**
- difficulty mix per theme: **5 low / 3 mid**
- side-to-move mix per theme: **4 white-to-play / 4 black-to-play**

The pool must be generated by a one-off script committed under `tools/`, reading a real Lichess puzzle CSV extract (or streaming the official dump), filtering to the 22 locked themes, stratifying by beginner-weighted bands (`600-999`, `1000-1299`), balancing side-to-move, and writing `js/puzzle-data.js` as a static committed file. No puzzle above `1299` belongs in the starter pool. The locked starter mix is **exactly 8 puzzles per theme** with **5 low / 3 mid** and **4 white / 4 black**. Extraction normalizes every row to the post-setup position: Lichess stores FEN before the opponent’s setup move, so the script applies that first move, stores the resulting `fen`, and trims `solution` so `solution[0]` is always the player’s first move.

### Reinforcement queue logic

Rule: failed puzzles are player-owned and theme-scoped. Next training sessions of that theme consume reinforcement first.

Pseudocode:

```text
onSessionEnd(theme, failedPuzzleIds):
  for each puzzleId:
    if not already queued:
      push { puzzleId, state: 'active', confirmations: 0 }

startSession(theme):
  take queued puzzles for theme first
  shuffle them
  if session still has empty slots:
    fill from unseen-or-random theme pool

onQueuedPuzzleSolved(theme, puzzleId):
  if state === 'active':
    state = 'pending-confirmation'
    confirmations = 1
  else if state === 'pending-confirmation':
    confirmations += 1
    if confirmations >= 2:
      remove from queue

onQueuedPuzzleFailed(theme, puzzleId):
  state = 'active'
  confirmations = 0
```

### Aptitude growth

Persist `training.aptitudes[theme]` as `0..100`. E.1 uses the reduced formula:

`aptitude = min(100, floor(solvedThemePuzzles * 1.5 + reinforcedResolves * 2))`

No `investedPoints` field exists in the save during E.1 or E.2. That term is added only in **E.3** when coaches and investment arrive:

`aptitude = min(100, floor(solvedThemePuzzles * 1.5 + reinforcedResolves * 2 + investedPoints * 6))`

### Test list

Target: **15-20 tests**.

- theme list returns the 22 locked keys
- unknown theme rejects cleanly
- self-training session starts with valid puzzles for theme
- seen puzzle IDs are persisted when a puzzle is shown
- reinforcement puzzles are selected before random theme puzzles
- failed puzzle enters queue once, no duplicates
- solving a queued puzzle moves it to `pending-confirmation`
- second successful solve in later session removes it
- failure during confirmation resets it to `active`
- aptitude increases after successful solves
- aptitude caps at 100
- `getTrainingBonusCount(theme)` starts at 0
- successful session grants exactly 1 training bonus charge for that theme
- failed session grants no bonus
- player state survives `init()` / save-load round-trip
- changing coach state later does not affect training persistence shape

### Explicit non-goals for E.1

- no coaches
- no finance deductions
- no in-game bonus invocation
- no temporary puzzle board inside live games
- no Flow bonus generation
- no reveal card
- no Stockfish playback reward
- no puzzle UI polish beyond debug/testability

## 4. E.2 design

**Q1 — Click handling during puzzle mode.** I pick **A**: `UIManager.onSquareClick` should branch early on a `_puzzleMode` flag and delegate to `BonusSystem.onPuzzleClick(square)`. This keeps one board click entry point, which matters because `UIManager` already centralizes read-only view mode, AI-thinking lockout, and turn ownership checks. The coupling is acceptable here: puzzle mode is not a separate screen, it is a transformation of the live game screen, so `UIManager` needs to know that the board is temporarily serving a different controller.

**Q2 — Board rendering source in puzzle mode.** I pick **B**: add a small `_pieceSource` state in `UIManager` with values `'live' | 'viewPly' | 'puzzle'`, then make `_getPieceAccessor()` switch over those three states. That extends the C.4 move-navigation pattern cleanly without changing `renderBoard()`'s signature or duplicating rendering logic. `BonusSystem` owns the temporary puzzle `Chess()` instance, but `UIManager` owns the board renderer, so the renderer should ask “which source am I drawing from?” rather than be passed ad hoc accessors.

**Q3 — Puzzle move validation.** I pick **A**: compare the player’s clicked move to the expected next UCI step from the stored solution sequence. On enter, `BonusSystem` loads the puzzle, stores `solutionIndex = 0`, and each completed click pair becomes a UCI string like `e2e4` or `e7e8q`; if it matches `solution[solutionIndex]`, the temporary puzzle chess instance advances and the pointer increments, otherwise the puzzle fails immediately. This is the simplest and most faithful approach because the Lichess data already gives us ordered moves, and it avoids SAN/FEN translation complexity in the first in-game iteration.

**Q4 — Focus pause during puzzle mode and playback.** I want the hybrid you anticipated. During the puzzle itself, **no explicit pause is needed** because the real game is frozen and the temporary `Chess()` instance never calls `ChessEngine.makeMove`, so `FocusSystem` naturally receives nothing. During the **post-success Stockfish playback**, we do need an explicit guard, because those moves will be played on the real game via normal APIs. So E.2 should add `FocusSystem.pauseForPlayback()` / `resumeFromPlayback()` and have `evaluateMoveDelta`, `apply`, and related reactive hooks short-circuit while playback is active.

**Q5 — Stockfish playback loop.** `BonusSystem` should own the playback loop, because it owns the reward budget, the paused-click window, and the success/failure transition; `UIManager` remains the executor of board moves and Maia replies. Pseudocode:

```text
invokeBonus(theme):
  consume charge, emit bonus_invoked
  snapshot real-game context
  enter puzzle mode

onPuzzleSolved():
  exit puzzle mode, show brief outcome card
  movesLeft = 2 + aptitudeBonus
  FocusSystem.pauseForPlayback()
  UIManager.lockInputForPlayback()
  while movesLeft > 0 and !ChessEngine.isGameOver():
    await delay(1100)
    best = await BonusSystem.getStockfishMoveFromCurrentPosition()
    if no best: break
    UIManager.applyPlaybackMove(best)
    if ChessEngine.isGameOver(): break
    movesLeft -= 1
    if movesLeft <= 0: break
    await UIManager.triggerAIMoveAndWait()
  UIManager.unlockInputAfterPlayback()
  FocusSystem.resumeFromPlayback()
  emit bonus_resolved
```

If the game ends mid-loop, playback stops cleanly and control never returns to “your turn.” If Maia’s reply or the player-side best move cannot be produced, the loop exits silently and returns control without retry/skip UI in E.2.

**Training bonus invocation UI placement.** E.2 uses **option A**: a permanent sidebar panel under `#stockfish-section`, titled **`Training bonuses`**. The panel is **hidden when inventory is empty**. The sidebar shows **one generic button per bonus source**, not one per theme: training bonuses render as `Training bonus (N)` where `N` is the total number of stored training charges across all themes. Clicking it invokes `invokeNextAvailableTrainingBonus()`, which picks the theme with the highest current charge count and breaks ties by alphabetical theme key. The actual theme is revealed only once puzzle mode opens, in the banner and outcome card. When invocation is temporarily illegal, the button stays **visible but disabled** so the player can still see that stock exists.

**Puzzle-mode board transformation.** E.2 must make puzzle mode unmistakable on the existing game screen. The board keeps the same rank/file labels and orientation logic, but the container gains a stronger accented border and a puzzle-mode tint on the board shell. A centered banner sits above the board, reading `PUZZLE — Fork — Solve to activate` for training bonuses, in large pixel text. On entry, the puzzle shows **no last-move highlight** from the real game and no inherited arrow/threat overlays; it is a fresh puzzle state, not a continuation of the live move history. Legal-move hints stay available only for the **currently selected piece**, using the same selected-piece interaction as live mode, because the player still needs normal board affordances to solve by clicks.

**Outcome card timing and placement.** For E.2 training bonuses, the outcome card is a **non-modal overlay** on the game screen, centered horizontally and placed just above / lightly overlapping the board. It auto-dismisses after **1500ms**, is not clickable, and uses the existing `--px-*` palette. The text is locked: success shows `Puzzle solved! +N Stockfish moves`; failure shows `Puzzle failed. No reward.` After the 1500ms, the card disappears and playback begins automatically on success, or control returns immediately to the player on failure.

**Auto-play of opponent replies.** Lichess puzzle solutions are full alternating lines, so E.2 must auto-play forced opponent replies on the temporary puzzle board instead of treating the sequence as “player moves only.” After a correct player click advances `solutionIndex`, if the next expected move belongs to the opponent, `BonusSystem` enters a non-clickable waiting sub-state and auto-plays that reply after `PUZZLE_OPPONENT_DELAY_MS = 500`. After **every** pointer increment, whether from the player or the auto-played opponent move, the system checks `solutionIndex >= solution.length`; puzzles may end on either parity, and mate-in-1 style single-move lines therefore terminate immediately with no auto-play.

**Adaptive puzzle rating and side matching.** E.2 selection is driven by a dedicated `training.puzzleRating` / `training.puzzleRatingRd` pair, not by player Elo. The locked starter parameters are **500 / 300**, giving an initial dynamic selection window of `max(150, round(rd * 1.2)) = 360`, which makes the first 10-15 puzzles feel easy before the range tightens toward `150` as RD converges to `50`. Successes and failures update this rating through an Elo-style expected-score formula with asymmetric failure impact (failures count at 50% weight), and in-game invocation prefers puzzles whose FEN turn color matches `ChessEngine.getPlayerColor()`, falling back only if that filtered pool is empty.

### `bonus-system.js` public API

- `BonusSystem.init()` — boot runtime state and event subscriptions.
- `BonusSystem.getInventory()` — expose persisted training bonus counts.
- `BonusSystem.canInvokeBonus(theme)` — true only if the player has a charge, it is their turn, and no puzzle/playback is active.
- `BonusSystem.invokeBonus(theme)` — consume a charge and enter puzzle mode.
- `BonusSystem.isInPuzzleMode()` — UI guard for board transformation.
- `BonusSystem.isPlaybackActive()` — UI/input guard during auto-play reward.
- `BonusSystem.getPuzzleState()` — current puzzle snapshot for rendering (`theme`, `fen`, `solutionIndex`, `selectedSquare`, `status`).
- `BonusSystem.onPuzzleClick(square)` — route square clicks while `_puzzleMode` is active.
- `BonusSystem.resolvePuzzleFailure()` — exit puzzle mode, emit failure, resume normal game.
- `BonusSystem.resolvePuzzleSuccess()` — exit puzzle mode and launch playback.
- `BonusSystem.getRewardMoveCount(theme)` — compute `2 + aptitudeBonus` for E.2.
- `BonusSystem._runPlayback(moveBudget)` — internal async loop over player auto-moves and Maia replies.

`cancelPuzzle()` should **not** exist in E.2. One-attempt, no-cancel keeps the commitment cost of using a bonus clear and avoids messy “refund or consume?” semantics.

### `puzzle-system.js` E.2 API extensions

These two methods are locked as canonical E.2 extensions. `BonusSystem` must call them instead of mutating `CareerManager.training` directly:

- `PuzzleSystem.pickInGamePuzzle(theme) -> puzzle | null`
  - reinforcement-first: if `training.reinforcementQueues[theme]` has entries, pick from there
  - otherwise: pick an unseen or rating-matched puzzle within `playerElo ± 200`
  - marks the chosen puzzle as seen before returning
  - persists via `CareerManager.save()`
  - returns `null` if no suitable puzzle exists
- `PuzzleSystem.consumeTrainingBonus(theme) -> boolean`
  - decrements `training.trainingBonuses[theme]` if at least one charge exists
  - returns `true` on success, `false` when empty
  - persists via `CareerManager.save()`

### `tests/bonus-system.test.js` plan

### Non-goals for E.2

- no Flow bonuses

### Top 3 risks

- `UIManager` state collisions between `_viewPly`, `_puzzleMode`, and playback locks
- async race conditions between the playback loop and Maia’s existing think/move path
- puzzle UX clarity: the board must feel unmistakably “not the real game” while still reusing the same renderer

## 5. Open questions parked for later sub-phases

- Exact training session length (`X puzzles per session`) can be tuned after E.1 playtests.
- Real board-based puzzle move validation belongs to the later training UI pass.
- Starting-profile bonus application belongs to E.2 reward resolution, not E.1.
- Coach-driven aptitude acceleration belongs to E.3 once `staff-system.js` exists.
- Flow-bonus reservation and hidden-theme reveal belong to E.4 only.

## 6. Risks

The biggest E.1 risk is overscoping the session model and accidentally building half of E.2. Keep `puzzle-system.js` pure and boring: selection, persistence, reinforcement, aptitude, bonus count. The second risk is data sparsity per theme; if the starter pool is too small, seen-puzzle and reinforcement behavior will look broken, so even a "small" Phase E dataset needs enough per-theme headroom to exercise the rules honestly.
